# CVE PoC 检索预研报告

## 1. 背景

当前项目已经落地了 3 种与 `CVE -> PoC线索` 相关的检索模式：

1. GitHub 检索
2. 中文互联网检索
3. 微信公众号归档检索

并提供了统一聚合入口：

- `cve_lookup_api.py`
- `cve_cn_search.py`
- `cve_wxvl_search.py`
- `cve_poc_search.py`

本报告目标是说明：

- 当前已经能做什么
- 进一步可以做什么
- 哪些方向实现成本高、稳定性差或暂时不建议做

## 2. 当前方案概览

### 2.1 GitHub 检索模式

脚本：

- `cve_lookup_api.py`

当前流程：

1. 通过 GitHub Search API 搜索包含指定 `CVE` 的仓库
2. 对仓库按相关性做二次打分
3. 返回更可能是真 PoC/Exploit 的前 3 个仓库
4. 查询 GitHub Advisory
5. 从 Advisory Description 和 References 中提取 PoC 信号

输出重点：

- `repo_search.top_repositories`
- `advisory.selected.likely_poc_references`
- `description_mentions_poc`

### 2.2 中文互联网检索模式

脚本：

- `cve_cn_search.py`

当前流程：

1. 生成一组中文查询词
2. 走搜索引擎或原生站内搜索
3. 合并结果
4. 对标题、摘要、来源进行规则打分
5. 过滤通告、预警、厂商公告、记录页

输出重点：

- `likely_cn_articles`
- `top_results`
- `fallback_results`

### 2.3 微信公众号归档检索模式

脚本：

- `cve_wxvl_search.py`

当前流程：

1. 优先读取 `gelusus/wxvl` 的 `data.json` 标题索引
2. 标题命中不足时，下载并缓存 `wxvl` 仓库快照
3. 从 Markdown 归档中回退检索正文
4. 过滤通告型公众号、公告型模板内容
5. 只保留更像分析/复现/PoC 的文章

输出重点：

- `wechat.articles`

### 2.4 统一聚合模式

脚本：

- `cve_poc_search.py`

当前统一输出结构：

```json
{
  "CVE-2024-3400": {
    "github": {
      "repos": [],
      "poc_references": []
    },
    "brower": {
      "likely_cn_articles": [],
      "top_results": [],
      "fallback_results": []
    },
    "wechat": {
      "articles": []
    }
  }
}
```

## 3. 当前已具备能力

### 3.1 已经能做的

#### A. 输入一个或多个 CVE，聚合多源 PoC 线索

已经支持：

- 单个 CVE
- 多个 CVE
- 从文件批量输入

适合作为人工研判前的第一层情报聚合。

#### B. GitHub 侧能输出“更像 PoC”的结果，而不是原始搜索页堆砌

当前不是简单把 GitHub 搜索结果原样返回，而是做了筛选：

- 优先命中仓库名/描述直接包含 `CVE`
- 提高 `poc` / `exploit` / `demo` 等关键词权重
- 降低 fork 干扰
- Advisory References 也会单独筛一次

这使得 GitHub 部分具备基础可用性。

#### C. 中文互联网结果已经具备一定“去通告化”能力

当前中文检索不是泛搜索，而是：

- 优先保留分析/复现类标题
- 厂商通告、公告、记录页降权或剔除
- 输出拆成高置信和低置信两层

这部分已经能满足“找分析/复现文章”的基础需求。

#### D. 微信公众号归档结果可作为中文 PoC 线索补充

`wxvl` 归档模式的价值不在于“全网搜索”，而在于：

- 某些公众号文章搜索引擎抓不到
- 标题索引查找很快
- 能补到中文互联网搜索漏掉的复现文章

#### E. 统一入口已具备可脚本化输出

统一入口输出是 JSON，便于后续：

- 接消息推送
- 接 Google Sheet
- 接 OpenClaw skill
- 接后续策略引擎

## 4. 当前存在的明确边界

### 4.1 GitHub 模式的边界

#### A. 当前检索范围还不够宽

当前 GitHub 模式本质上是：

- 搜索仓库
- 搜索 advisory references
- 按规则判断“疑似 PoC 链接”

它当前更擅长发现：

- 仓库名、描述、README 中直接暴露 `CVE` 的仓库
- GitHub Advisory `references` 中外链出来的 PoC

它当前还不擅长系统性发现：

- 仓库 `Issues` 里的 PoC 讨论
- `Discussions` 里的 PoC 线索
- 仓库代码内容里提到 `CVE`，但仓库名和 README 不明显的项目

也就是说，当前 GitHub 模式更像“仓库级 PoC 发现 + Advisory 引用筛选器”，
还不是完整的“GitHub 全域 PoC 检索器”。

#### B. 只能发现“公开线索”，不能证明“PoC 真可用”

即使查到了 GitHub 线索，它也不能直接证明：

- 仓库能跑
- PoC 有效
- 代码不是误导/钓鱼/无关脚本

因此当前更适合作为“线索发现”，不适合作为“最终验证”。

#### C. 规则筛选天然存在误报和漏报

例如：

- 一些研究文章链接会被当作 PoC 线索
- 一些 PoC 仓库如果命名隐蔽，会被漏掉

### 4.2 中文互联网模式的边界

#### A. 搜索结果受搜索后端影响很大

当前中文模式不掌握搜索引擎，只是消费搜索结果。

因此会受以下因素影响：

- 搜索引擎索引新鲜度
- 风控/反爬
- query 质量
- 地域差异

#### B. 互联网搜索可用，不等于原生站内搜索可用

当前比较容易支持、已经有一定可用性的来源：

- `research.qianxin.com`
  - 原因：站内搜索结果结构相对稳定，适合原生解析
- `cloud.tencent.com`
  - 原因：页面里有可提取的数据结构，适合做轻量检索
- 搜索引擎能直接返回的普通博客/聚合页
  - 例如部分 `cn-sec`、`secrss`、普通技术博客页
- `先知社区`、`安全客` 等社区文章页
  - 说明：如果搜索引擎已经把文章 URL 返回出来，这类页面可以作为结果页使用
  - 限制：它们不适合作为当前阶段的“原生站内搜索入口”

当前比较难稳定支持的来源：

- `FreeBuf`
  - 问题：容易遇到 WAF/反爬，普通请求不稳定
- `博客园`
  - 问题：常见人机验证，不适合简单脚本抓取
- `CSDN`
  - 问题：搜索结果页前端渲染较重，直接请求拿不到稳定结果
- `安全客`
  - 问题：如果要求做原生站内搜索，结果结构不稳定，当前未找到长期可靠的数据接口
- `先知社区`
  - 问题：如果要求做原生站内搜索，搜索页不适合直接解析，真实 query 结果不稳定
- `Seebug`
  - 问题：请求慢，且可用性不稳定
- `SecurityOnline`、`GBHackers`
  - 问题：RSS/搜索抓取容易被 Cloudflare 或 WAF 拦截

所以当前中文互联网模式不是“任意中文站点都能稳定接入”，而是：

- 对少数结构稳定站点做原生适配
- 对大量站点优先依赖搜索引擎返回的结果页
- 对反爬重、动态化强的站点，不强求原生站内搜索能力

#### C. 现在更像“高价值结果筛选器”，不是“全文抓取器”

当前中文模式主要使用：

- 标题
- 摘要
- 来源

并不是逐篇抓正文深度解析，因此仍存在：

- 标题像复现，正文其实是通告
- 摘要太短，判断不充分

### 4.3 微信公众号模式的边界

#### A. 依赖 `wxvl` 数据源完整性

它不是实时抓公众号，而是依赖归档。

所以它能做的是：

- 在 `wxvl` 已收录的前提下做检索

做不到的是：

- 保证覆盖所有公众号
- 保证实时性

#### B. 公众号来源判断只能做规则近似

当前通过：

- 标题关键词
- 正文模板
- 来源黑名单

来剔除通告型文章。

这种方式有效，但不可能完全准确。

## 5. 可以继续做的方向

### 5.1 低成本、收益高的方向

#### A. GitHub 模式补缓存

可以做：

- 对同一个 `CVE` 的 GitHub 查询结果做本地缓存
- 对 Advisory 查询做短期缓存

收益：

- 降低 GitHub API 压力
- 减少 rate limit
- 降低重复查询耗时

#### B. 中文模式补正文二次抓取

可以做：

- 只对 `likely_cn_articles` 前几条结果抓正文
- 再用正文做一次复现/分析判断

收益：

- 降低“标题看着像复现，正文其实是通告”的误报

#### C. 微信模式补来源配置外置

当前公众号来源黑名单和公告模板是写在代码里的。

可以做：

- 抽成 YAML/JSON 配置

收益：

- 不需要每次改代码
- 方便持续维护黑名单

### 5.2 中成本、值得做的方向

#### A. 中文模式增加更多原生站点适配

可以继续补：

- 安全客
- 先知
- Seebug
- 其他中文安全媒体或社区

前提：

- 找到稳定可解析的真实数据源

收益：

- 提升覆盖率

#### B. 统一打分体系

目前三种模式都有自己的规则。

可以做：

- 统一 `confidence score`
- 统一结果分层
- 统一 `reason` 字段

收益：

- 后续更容易接自动化决策
- 结果解释性更强

#### C. 增加去重和跨源合并

现在同一个 PoC 线索可能在：

- GitHub
- 中文网站
- 微信文章

重复出现。

可以做：

- 标题归一化
- URL 去重
- 跨源聚合成一条“线索实体”

收益：

- 结果更清爽

## 6. 能做但暂时不好做的方向

### 6.1 全自动稳定抓所有中文站点

不好做的原因：

- 各站点反爬策略差异很大
- 很多搜索页依赖 JS
- 数据结构经常变化
- 部分站点需要登录、人机验证或 Cloudflare challenge

这类工作会把系统从“检索器”变成“重型爬虫系统”，维护成本显著上升。

### 6.2 只靠规则实现“零误报”

不好做的原因：

- 通告和分析文在标题上可能极其相似
- 有些文章标题写“已复现”，正文仍然只是转载通报
- 不读正文、不做人工确认，就很难零误报

因此这部分更合理的目标是：

- 降误报
- 提高高置信结果比例

而不是承诺零误报。

### 6.3 自动验证 PoC 是否可利用

理论上可以做，但当前阶段不建议。

原因：

- 要搭靶场
- 要区分 HTTP / 非 HTTP 漏洞
- 要处理 payload 副作用
- 很容易越过“情报聚合”边界，进入“自动利用验证”系统

这会显著提高复杂度和风险。

## 7. 当前最不建议做的事情

### A. 为所有站点统一上浏览器自动化

这看起来能解决很多反爬问题，但代价过高：

- 慢
- 重
- 不稳定
- CI/云端环境维护麻烦

浏览器模式适合作为少数难点源的补充，不适合作为默认方案。

### B. 直接把结果当最终结论

当前系统的更合理定位是：

- PoC 线索发现器
- 多源情报聚合器

不是：

- 最终漏洞验证平台
- 最终情报定级平台

## 8. 推荐的后续落地方向

建议按下面顺序推进：

1. 给 GitHub 模式加缓存和更稳定的错误降级
2. 给中文模式加“高置信结果正文二次抓取”
3. 把微信模式的黑名单/模板规则配置化
4. 做统一打分和跨源去重
5. 最后再考虑少量站点的浏览器兜底

这个顺序的优点是：

- 改动小
- 回报快
- 不会过早引入高维护成本模块

## 9. 总结

当前三种模式已经具备实际使用价值：

- GitHub 模式适合发现公开仓库和 advisory 引用中的 PoC 线索
- 中文模式适合发现博客、论坛和安全分析文章
- 微信模式适合补充公众号归档中的中文 PoC 文章

统一入口已经能把三种结果合并成结构化 JSON，这一点非常关键，说明系统已经从“单点脚本”进入“可集成组件”阶段。

但同时也要明确：

- 它当前擅长的是“线索聚合”
- 不擅长的是“最终确认”
- 真正困难的部分不是再加几个源，而是如何在覆盖率、稳定性、误报率之间取得平衡

如果后续继续推进，优先级应该放在：

- 缓存
- 正文二次判定
- 规则配置化
- 跨源去重

而不是直接把系统做成重型爬虫或自动利用验证器。
